# dm.ai — AI-Powered Dungeon Master Toolkit

> AI proposes. You decide. Your world stays yours.

## What it is

dm.ai is an AI assistant for tabletop RPG game masters. It is not a replacement
for the DM — it is a force-multiplier that eliminates the tedious parts of session
prep while leaving all creative authority in human hands. When you describe a
narrative moment ("the players arrive at a coastal trading town"), dm.ai generates
a complete town: layout, notable buildings, named NPCs with personality traits and
dialogue hooks, local rumors, and a battle-map grid. Everything arrives as a
**Proposal** that you review, edit, and accept or reject. Nothing writes to your
campaign database without your explicit approval.

The philosophy is **AI proposes, DM decides**. The orchestration layer (Claude
Sonnet) handles narrative reasoning and world-building proposals. A faster model
(Claude Haiku) handles high-volume tasks such as NPC dialogue options and session
summaries. All generated content is indexed in a pgvector RAG pipeline so
subsequent proposals stay consistent with your established lore: a town built on
fishing will not later produce a landlocked cattle market, and an NPC you marked as
distrustful of strangers will not suddenly be warm and forthcoming in a later
session.

The MVP targets **D&D 5.5e**. The architecture is rule-system-agnostic by design:
a `RuleEngine` abstract base class in the `game-engine` package isolates all
system-specific logic, and new rule systems (Pathfinder, Call of Cthulhu, etc.)
can be added by subclassing it without touching the API or UI layers.

## Features

- **World Builder** — generates locations (realm → town → building → room
  hierarchy), complete with lore, history, and map data
- **Character Generator** — produces full stat blocks, personality traits, ideals,
  bonds, and flaws for PCs and NPCs
- **NPC Dialogue Agent** — returns three contextually appropriate dialogue options
  (warm / neutral / guarded) for any NPC
- **Combat Advisor** — suggests the tactically sound action + flavor text for each
  monster turn; DM approves before resolution
- **Battle Map** — interactive react-konva canvas with token drag-and-drop, fog of
  war, zoom/pan, and grid snap
- **Proposal Flow** — every AI-generated object is surfaced as a reviewable card;
  DMs can accept as-is, edit fields, or reject with notes
- **World Consistency RAG** — pgvector embeddings on worlds, locations, and
  characters prevent contradictions across sessions
- **Session Summaries** — end-of-session auto-summary generated by the fast model
  and stored for continuity
- **Real-time Updates** — WebSocket broadcasting keeps multiple browser tabs in
  sync during live sessions
- **Dual AI Backend** — use the Anthropic API with an API key, or the `claude` CLI
  for local/offline use

## Architecture

```
  dm-ui  (React 19 + Vite + react-konva)
  ├── ChatPanel  BattleMap  ProposalCard  CombatTracker
  └── Zustand store  ←  WebSocket hook
          │
          │  REST + WebSocket  (prefix: /api)
          ▼
  dm-api  (FastAPI + Python 3.12 + SQLAlchemy async)
  ├── REST routes: /worlds  /sessions  /characters  /locations  /combat  /ai
  ├── WebSocket:   /ws/sessions/{id}
  ├── AI layer
  │   ├── DMOrchestrator  (stateless, session-scoped)
  │   ├── AIBackend ABC
  │   │   ├── AnthropicBackend  (Anthropic SDK)
  │   │   └── ClaudeCLIBackend  (subprocess, claude CLI)
  │   └── Prompts / system prompt builder
  ├── PostgreSQL 16 + pgvector  (7 tables, vector(1536) embeddings)
  └── Redis  (pub/sub for WS broadcast)
          │
          ▼
  game-engine  (installable Python package, no FastAPI dep)
  ├── RuleEngine ABC  (interface.py)
  ├── types package  (enums + dataclasses)
  └── rules/dnd_5_5e/  (DnD55eEngine + data files)
          │
          ▼
  Anthropic Claude API  or  claude CLI
```

## Quick Start

### Prerequisites

- Docker + docker-compose (any recent version)
- Python 3.12+ (for local game-engine / dm-api development)
- Node 22+ (for local dm-ui development)

### Running with Docker

```bash
cp .env.example .env
# Edit .env — set ANTHROPIC_API_KEY, or set AI_PROVIDER=claude_cli
docker-compose up
```

- DM Dashboard: http://localhost:5173
- API + interactive docs: http://localhost:8000/docs

The `api` container waits for `postgres` and `redis` health checks before
starting. Alembic migrations run automatically on first start.

### Running locally (development)

#### Backend (dm-api)

```bash
cd dm-api
pip install -e "../game-engine"   # game-engine must be installed first
pip install -e ".[dev]"
alembic upgrade head              # requires running Postgres
uvicorn dm_api.main:app --reload --port 8000
```

#### game-engine tests

```bash
cd game-engine
pip install -e ".[dev]"
pytest tests/ -v
```

#### Frontend (dm-ui)

```bash
cd dm-ui
npm install
npm run dev
```

## Configuration

All variables are read from `.env` via pydantic-settings. Copy `.env.example` and
fill in the values relevant to your setup.

| Variable | Default | Description |
|---|---|---|
| `AI_PROVIDER` | `anthropic` | `anthropic` (API key) or `claude_cli` (local CLI) |
| `ANTHROPIC_API_KEY` | — | Required when `AI_PROVIDER=anthropic` |
| `ORCHESTRATOR_MODEL` | `claude-sonnet-4-6` | Main chat / proposal model |
| `PLANNING_MODEL` | `claude-sonnet-4-6` | World-building and complex reasoning |
| `GENERATION_MODEL` | `claude-haiku-4-5-20251001` | Fast tasks: summaries, dialogue |
| `DATABASE_URL` | `postgresql+asyncpg://dmuser:dmpass@localhost:5432/dmdb` | PostgreSQL (asyncpg) |
| `REDIS_URL` | `redis://localhost:6379` | Redis connection |
| `SECRET_KEY` | `dev-secret-key` | FastAPI signing key (change in production) |
| `FRONTEND_URL` | `http://localhost:5173` | CORS allowed origin |
| `CONTEXT_TOKEN_LIMIT` | `180000` | Token count that triggers context summarization |
| `CONTEXT_PRESERVE_LAST_N` | `5` | Messages kept verbatim before the summary |

## Development

AI agents (Claude Code and similar) should read **AGENTS.md** first — it contains
non-negotiable coding rules: no raw string literals for game concepts (use enums),
no `dict` parameters for structured entities (use dataclasses), 400-line file
limit, and the reviewer checklist.

Human contributors should read **CONTRIBUTING.md** for prerequisites, local setup,
test commands, PR process, and style guide.
